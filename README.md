# Исследование и автоматизация доступа к сервису Росреестра

**Сервис:** Справочная информация по объектам недвижимости в режиме online  
**URL:** https://lk.rosreestr.ru/eservices/real-estate-objects-online  
**Дата исследования:** март 2026

---

## 1. Цель

Автоматизировать получение справочной информации об объектах недвижимости из публичного сервиса Росреестра. Входные данные — диапазон кадастровых номеров или адреса квартир одного дома. Выходные данные — CSV-файл с характеристиками объектов.

---

## 2. Исследование сервиса

### 2.1 Общая структура

Сервис реализован как SPA (Single Page Application) на React. Фронтенд загружается с `lk.rosreestr.ru`, все данные получает через внутренний REST API по пути `/account-back/`.

Сайт использует российский удостоверяющий центр (УЦ), поэтому при открытии через браузер без установленного российского корневого сертификата возникает SSL-ошибка. На стороне скрипта эта проблема решается параметром `verify=False` в `httpx`.

### 2.2 Механизмы защиты

| Механизм | Описание |
|---|---|
| **Капча** | Требуется для каждого запроса данных объекта неавторизованным пользователем |
| **Авторизация** | Авторизованные пользователи (через Госуслуги) работают без капчи, но с лимитом запросов |
| **SSL** | Российский УЦ — не распознаётся браузерами без установленного сертификата |

### 2.3 Исследование сетевых запросов

Перехват сетевого трафика выполнялся через XHR-интерцептор, установленный в браузер через расширение Claude in Chrome. Было выполнено несколько поисковых запросов с ручным вводом капчи для захвата всех типов запросов.

**Все обнаруженные эндпоинты:**

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/account-back/captcha.png` | Получить изображение капчи |
| `GET` | `/account-back/captcha/{code}` | Валидация введённого кода |
| `GET` | `/account-back/address/search?term=...&objType=` | Автодополнение адреса |
| `POST` | `/account-back/on` | Получение данных объекта |
| `POST` | `/account-back/on/with-addresses` | То же, но с адресами (требует авторизации, 401) |
| `GET` | `/account-back/config` | Конфигурация приложения |
| `GET` | `/account-back/dictionary/{NAME}?sortKey=code` | Справочники (типы объектов, категории земель и т.д.) |

### 2.4 Детали ключевых запросов

**Получение данных объекта — `POST /account-back/on`:**

```http
POST /account-back/on
Content-Type: application/json

{
  "filterType": "cadastral",
  "cadNumbers": ["50:20:0010203:101"],
  "captcha": "r4rk7"
}
```

**Пример ответа (сокращённо):**

```json
{
  "elements": [{
    "cadNumber": "50:20:0010203:101",
    "cadQuarter": "50:20:0010203",
    "status": "1",
    "objType": "002001003000",
    "area": "38.50",
    "floor": null,
    "purpose": "206001000000",
    "address": {
      "region": "Московская обл.",
      "street": "Центральная",
      "streetType": "ул",
      "house": "12",
      "houseType": "д",
      "apartment": "47"
    },
    "cadCost": "4250000.00",
    "cadCostDate": null,
    "regDate": 1609459200000,
    "rights": [...],
    "encumbrances": [...]
  }]
}
```

**Валидация капчи — `GET /account-back/captcha/{code}`:**

- `200 OK` — капча верна, можно делать запрос данных
- `403 {"error": "Wrong captcha"}` — капча неверна, нужно повторить

### 2.5 Логика работы с капчей

Важное наблюдение: капча привязана к **сессии**, а не к конкретному запросу. Алгоритм следующий:

1. `GET /account-back/captcha.png` → новое изображение + код сессии сохраняется в cookie
2. `GET /account-back/captcha/{введённый_код}` → сервер проверяет введённый код против кода в сессии
3. `POST /account-back/on` с телом `{"captcha": "тот_же_код"}` → данные объекта

## 3. Исследование капчи

### 3.1 Характеристики

На основании визуального анализа 4 образцов:

- **Размер изображения:** ~266×58 px
- **Длина:** ровно 5 символов
- **Алфавит:** строчные латинские буквы (`a-z`) + цифры (`0-9`)
- **Помехи:** один диагональный штрих поверх текста; градиентный серый фон
- **Цвет текста:** серый или синий (варьируется)

Примеры из перехваченного трафика: `r4rk7`, `m3ynm`, `hebx4`, `f6nb2`, `8yn46`, `rpbed`, `hbkfn`.

### 3.2 Оценка сложности

Капча относится к классическому типу текстовых CAPTCHA низкой сложности. Основная помеха — одна линия — легко устраняется при бинаризации изображения. Специализированные OCR-библиотеки для капч решают её без дообучения.

### 3.3 Рассмотренные варианты решения

| Вариант | Плюсы | Минусы |
|---|---|---|
| **2captcha / anti-captcha** | Простая интеграция, высокая точность | ~$1 за 1000 капч, зависимость от внешнего сервиса |
| **EasyOCR** | Бесплатно, локально | Несовместимость с NumPy 2.x, тяжёлые зависимости (~500 МБ) |
| **ddddocr** | Бесплатно, локально, легковесный, создан специально для капч | — |

**Выбор: `ddddocr`** — библиотека, специализированная для простых текстовых капч, работает через ONNX Runtime, не требует scipy/numpy, устанавливается одной командой.

### 4.2 Поток данных

```
Входные данные (список кадастровых номеров)
    │
    ▼
для каждого номера:
    │
    ├─ GET /account-back/captcha.png  ──▶  PNG-изображение
    │                                          │
    │                                          ▼
    │                                      ddddocr.classification()
    │                                          │
    │                                          ▼
    ├─ GET /account-back/captcha/{code}  ──▶  200 OK / 403
    │       (валидация)                        │
    │                                    ┌─────┴──────┐
    │                                  верно        неверно
    │                                    │          (повтор)
    │                                    ▼
    ├─ POST /account-back/on  ──────▶  JSON с данными объекта
    │
    ▼
flatten_object()  →  плоский словарь
    │
    ▼
CSV (utf-8-sig для Excel)
```

### 4.3 Обработка ошибок и ретраи

При каждом запросе объекта выполняется до `MAX_RETRIES = 5` попыток. Повтор происходит при:
- OCR не смог распознать капчу (результат не 5 символов)
- Сервер вернул 403 на валидацию капчи
- Сетевая ошибка / таймаут

### 4.4 Поля в выходном CSV

| Поле | Описание | Пример |
|---|---|---|
| `cad_number` | Кадастровый номер | `50:20:0010203:256` |
| `cad_quarter` | Кадастровый квартал | `50:20:0010203` |
| `status` | Статус объекта (текстовый) | `Актуально` |
| `obj_type` | Вид объекта недвижимости (текстовый) | `Помещение` |
| `area` | Площадь, кв.м | `38.5` |
| `floor` | Этаж | `5` |
| `purpose` | Назначение (текстовый) | `Нежилое` |
| `address_full` | Полный адрес из поля `readableAddress` API | `Московская обл., г. Химки, ул. Центральная, д. 12, кв. 15` |
| `address_region` | Регион | `Московская обл.` |
| `address_street` | Улица | `Центральная` |
| `address_house` | Номер дома | `12` |
| `address_apartment` | Квартира / помещение | `15` |
| `reg_date` | Дата присвоения кадастрового номера | `01.01.2021` |
| `cad_cost` | Кадастровая стоимость, руб | `4250000.00` |
| `cad_cost_date` | Дата определения кадастровой стоимости | `01.01.2024` |
| `rights` | Права: тип, номер, дата, доля; через `;` | `Собственность № 50:20:...-3 от 15.03.2021; ...` |
| `encumbrances` | Обременения: тип, номер, дата; через `;` | `Аренда № 50:20:...-7 от 20.05.2024` |
| `raw_json` | Полный ответ API в JSON (для разбора дополнительных полей) | `{...}` |

Коды статусов, типов объектов и назначений декодируются в текст по встроенным справочникам. Для помещений этаж берётся из поля `levelFloor`, для зданий — из `floor`.

## 5. Установка и запуск

### 5.1 Установка зависимостей

```bash
pip install -r requirements.txt
```

Или вручную:

```bash
pip install httpx ddddocr
```

### 5.2 Запуск

```bash
python rosreestr_parser_ddddocr.py [OPTIONS]
```

### 5.3 Аргументы командной строки

| Аргумент | По умолчанию | Описание |
|---|---|---|
| `-i FILE`, `--input FILE` | встроенный список | Текстовый файл с кадастровыми номерами (по одному на строке) |
| `-o FILE`, `--output FILE` | `results.csv` | Имя выходного CSV-файла |
| `-r N`, `--retries N` | `5` | Максимальное число попыток при ошибке капчи |
| `-s`, `--silent` | выключен | Тихий режим: ничего не выводить в консоль |
| `-l FILE`, `--log FILE` | нет | Файл для записи логов с таймстемпами |

### 5.4 Примеры запуска

```bash
# Базовый запуск (встроенный список номеров, вывод в results.csv)
python rosreestr_parser_ddddocr.py

# Входной файл и именованный выходной файл
python rosreestr_parser_ddddocr.py -i input.txt -o dom_12.csv

# Тихий режим с логированием в файл (для cron / фонового запуска)
python rosreestr_parser_ddddocr.py -i input.txt -s -l run.log

# Вывод в консоль и одновременно в лог-файл
python rosreestr_parser_ddddocr.py -i input.txt -l run.log

# Увеличить число попыток распознавания капчи
python rosreestr_parser_ddddocr.py -i input.txt -r 10
```

### 5.5 Формат входного файла

Текстовый файл, по одному кадастровому номеру на строке:

```
50:20:0010203:100
50:20:0010203:101
50:20:0010203:102
```

### 5.6 Формат лог-файла

При указании `-l` лог пишется в формате:

```
2026-03-14 12:34:56  INFO      [1/10] 50:20:0010203:256
2026-03-14 12:34:57  DEBUG     Капча: 'r4rk7' (попытка 1)
2026-03-14 12:35:00  INFO        ✓ Московская обл., г. Химки, ул. Центральная, д. 12, кв. 15 | 38.5 м²
2026-03-14 12:35:15  WARNING     ✗ не найден: 50:20:0010203:199
2026-03-14 12:35:40  INFO      Готово: 9/10 объектов → dom_12.csv
```

В консоль (без `-s`) выводится только текст сообщений, без времени и уровня.

## 6. Ограничения и риски

- **Точность OCR:** `ddddocr` показывает высокую точность на простых текстовых капчах, но гарантий нет. При частых ошибках — возможен fallback на 2captcha или сбор датасета для дообучения.
- **Лимиты сервера:** Неизвестны скрытые rate-limiting механизмы. Задержка `REQUEST_DELAY = 3 сек` снижает риск блокировки.
- **Изменение API:** Сервис государственный — структура ответа и эндпоинты могут измениться без предупреждения. Поле `raw_json` в CSV позволяет разобрать данные заново без повторного парсинга.
- **SSL:** Скрипт использует `verify=False` — достаточно для задачи, но не рекомендуется в production-окружениях с повышенными требованиями к безопасности. Альтернатива — установить российский корневой сертификат.
